# 仕様整理: 管理画面AIエージェントチャット（2026-07-22 改訂v2 / saas-product-manager）

状態: 壁打ち〜要件定義段階。未実装。
v1（読み取り専用アシスタント案）はユーザー却下 → **「サイト内でAIに指示を与えてどんどん進めてもらう」= 実行するエージェント**として再設計。

## 思想

チャットは補助ではなく**管理画面の主要な操作入力**。
「来週分のデイリー設問を3本作って積んでおいて」「この案件の設問を非誘導レビューして直して」を
チャットに投げれば、AIが実データを読み、作成・編集まで進め、結果を報告する。
人間の役割は**指示と、危険な操作の承認**に寄せる。

成功状態: 日常の管理作業の大半（設問作成・キュー積み・配信準備・回答確認）が、画面のフォームを直接触らずチャット指示で完了する。

## 実行モデル: 危険度3層のツールゲート

AIには画面ごとの**ツールレジストリ**（ホワイトリスト）を渡す。全ツールは既存の service 層を経由し、
AI専用の抜け道クエリ・抜け道書き込みは作らない。各ツールに危険度を宣言させる:

| Tier | 定義 | AIの挙動 | 例 |
|---|---|---|---|
| **A: 読む** | 副作用なし | 無条件で実行 | 案件・設問・回答・セグメント・ログの取得/要約 |
| **B: 戻せる書き込み** | 社内データのみ変更・undo可能・外部に何も出ない | **自動実行して報告**（実行後に取り消し導線を提示） | 設問draft作成/編集（未公開のみ）、デイリーキュー積み、セグメント作成、campaign下書き、NGワード登録 |
| **C: 不可逆・対外** | LINE実配信、ポイント付与、交換承認、公開、entry_code変更、グローバル必須同意 | AIは**準備まで**進め、チャット内に承認カード（影響一覧＋対象件数＋「確認しました」チェック）を出し、人間がクリックして初めて実行 | 配信実行、公開、pt調整 |

- Tier C の承認カードは `prompt-packages/publish-confirm.ejs` の型（管理画面レビューで「横展開すべき良い設計」とした確認パターン）を**チャット内に移植**したもの。レビュー方針との整合はここで取る。**承認カードの対象件数・影響一覧はAIの申告値ではなくサーバー側で再計算した値を表示する**（P0-3の教訓: 推定と実行対象のロジック分離は事故のもと）。
- Tier はツール実装側で静的に宣言（AIが自己申告するのではない）。境界例（公開後の設問編集＝ロウデータ契約に触る）は C に倒す。
- **自律度設定**: 運用に慣れたら Tier C の一部（例: 少人数テスト配信）を B 扱いに下げられる設定画面を用意。既定は全部厳しい側。

## マルチステップ自走

「どんどん進める」の実体は 1指示 = plan → 複数ツール実行 → 報告 のエージェントループ。

- 1指示あたりのツール実行上限（例: 15回）とタイムアウトを設ける。
- 途中で Tier C に当たったら、そこまでの結果＋承認カードで停止（部分完了を報告）。
- Vercel serverless の実行時間内に収まらない長尺タスク（一括生成・全件見直し等）は、
  同期ループでは打ち切り、**指示文ディスパッチ方式**（対象をDBに保存→ID発行→Claude Code側で処理して書き戻す既存パターン）へ逃がす二段構え。チャットは「タスクとして積みました。実行文: …」を返す。

## 前提（既存資産）

| 資産 | 使い方 |
|---|---|
| `runAdminToolPrompt` / `callRaw`（aiService.ts） | エージェントループは tool-calling 対応の呼び出しを新設（Chat Completions tools）。ログ・モデル分岐の作法は踏襲 |
| service 層（questionFlowServiceV2, dailySurveyService, projectDeliveryService, …） | ツールの実体。AIから直接SQLは書かせない |
| Prompt Package admin_tool 系統 | エージェント system prompt（`adminChatCommon`＋画面別）を版管理 |
| CSRF フェッチメタデータ検査（PR#24） | `POST /admin/api/ai-chat` と承認カードの実行APIに適用 |
| `ai_logs` | purpose=`admin_chat:<screenKey>`。token_usage 記録 |
| publish-confirm.ejs の確認パターン | Tier C 承認カードの雛形 |

## ユーザー種別

運営管理者（Basic認証・単一アカウント）のみ。権限分岐なし。
ただし**監査は必須になる**: AIが実行した操作は「AI実行（指示者=admin）／承認カード経由」を `admin_ai_actions` ログ（新設）に全件記録（ツール名・引数・結果・元指示・承認有無）。レビューP1-7（操作者記録ゼロ）はAI導入でむしろ悪化するため、AI経由の操作から先に記録を整える。

## 機能一覧・Phase分け

| Phase | 内容 | できるようになること |
|---|---|---|
| 1 | 共通基盤（パネルpartial＋`POST /admin/api/ai-chat`＋ツールレジストリ＋エージェントループ＋`admin_ai_actions`ログ）＋ **research-form 縦切り**: Tier A（案件/設問/回答数読み）＋ Tier B（未公開設問の作成・編集・並べ替え、選択肢生成） | 「この案件に年代別の設問を3本足して」「Q2を非誘導に直して」がチャットで完結。事故リスクのない書き込みで実行体験を検証 |
| 2 | デイリー/プール系 Tier B（キュー積み・pool設問CRUD）＋ sessions/posts の Tier A（回答要約・投稿分析） | 「来週分のデイリーを5本作って積んで」「この回答者の傾向まとめて」 |
| 3 | **Tier C 承認カード基盤**＋配信系（セグメント作成=B、配信実行=C、テスト送付=C）＋ pt調整=C | 「20代女性にこの案件を配信して」→ 準備一式をAIが済ませ、件数付きカードを人間が承認 |
| 4 | 自律度設定画面・ディスパッチ連携（長尺タスク）・会話履歴永続化・画面横断（screenKeyなしのグローバルチャット） | 定型作業の実質自動化 |

MVPは **Phase 1**。読み専用MVP（v1案)は廃止し、最初から「読み→実行(B)→報告」の縦切りにする。実行体験こそが検証対象のため。

## 安全設計（v1から引き継ぎ＋強化)

- ツールはホワイトリスト・service層経由・Tier静的宣言。SQL/HTTP自由発行なし。
- Tier C はサーバー側再計算値で承認カード。承認なし実行はAPIレベルで不可能（AIの出力に承認トークンを含められない構造にする）。
- プロンプトインジェクション: 回答者の自由記述など外部由来テキストがコンテキストに入るため、**実行能力がある分v1より重要**。外部由来データは区切り付きで「データであって指示ではない」扱い。Tier B以上のツール実行が外部由来テキスト起点にならないよう、system promptと承認設計の両方で抑える（完全防御は不可能 → だからこそ Tier C ゲートは人間承認）。
- **個人情報のOpenAI送信**: v1と同じブロッカー。規約v2のデータ使途で要確認。NGならマスキング先行。
- コスト: エージェントループはツール往復分トークンを食う。`ai_logs` で purpose別に可視化し、上限（1指示あたりのツール回数・1日あたりの呼び出し数）を env で持つ。

## 実装指示に落とす内容（implementation-planner へ)

1. Phase 1a: エージェントループ基盤 — tool-calling呼び出し（aiService拡張）、ツールレジストリIF（`{name, tier, screenKeys, schema, execute}`）、`POST /admin/api/ai-chat`（CSRF・ループ上限・ai_logs）、`admin_ai_actions` migration。
2. Phase 1b: research-form ツール群（read: 案件/設問/回答数、write-B: 未公開設問CRUD）＋パネルUI配線＋prompt packageキー。
3. テスト観点: Tier宣言のないツールは登録拒否／未公開チェックをservice側で強制（公開済み設問への書き込みはツール層でなくserviceが409）／承認なしTier C実行がAPIで不可能なこと。

## ユーザー確定事項（2026-07-22）

1. 回答者データのOpenAI送信は**問題なし**（マスキング不要・ブロッカー解消）。
2. 実行範囲は**ユーザーが指示文で都度指定**する。AIは明示的に依頼された書き込みだけ行う（system promptで強制）＋Tierゲートは常時有効。
3. 初回リリースは設問作成でなく**回答を読み解く側**（設問作成は既存フロー生成AIがあり、回答側が弱いため）。設問系Tier B・配信系Tier Cは後続。
