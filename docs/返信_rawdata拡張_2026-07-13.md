## 2. RANK の確定事項（そちらの未解決 §13-6 への回答）

**出力時点の現在ランクを出す。回答完了時点の復元はしない。**

- 理由: `respondent_rank_histories` を書いているのはレガシーの `rankService`（`respondents.total_points` 基準）。
  ポイントの正準はすでに `user_points` に移行済みで、履歴から「完了時点」を復元しても**復元元が不整合**。
  精度が上がらない。会員本人がマイページで見ているランク（`user_ranks`）と一致させる方が意味論的に正しい。
- **影響**: 同じ wave を後日再エクスポートすると RANK が変わりうる（**非再現**）。
  定点比較で版を固定したい場合は、**エクスポートした CSV そのものを保管**して使い回すこと
  （Hibi 側は `export_jobs` に出力履歴を残しているので、いつ出したものかは追跡できる）。
- rank→重みの写像はそちらのレシピ設定のままでよい。値は `rank_code`（`bronze` / `silver` / … の文字列）。
  layout の `RANK` 行にコード→表示名の対応表を出している。

## 3. 【要注意】SURVEY_VERSION は当面ほぼ空欄になる

実装は入ったが、**`questionnaire_snapshots` が本番にまだ1件も無い**（＝調査票スナップショットの確定運用が始まっていない）。
そのため現時点では:

- `SURVEY_VERSION`（行側）: **全行空欄**。`sessions.snapshot_id` が記録されるのは、スナップショット確定後に**開始された**回答から。
- `question_version`（layout側）: 有効スナップショットが無いので**空欄**。

**過去の回答に遡って版数を埋めることはできない**（どの定義で答えたかの記録が存在しないため。推測で埋めるのは
データを偽ることになるのでやらない）。Hibi 側で「送付前にスナップショットを確定する」運用を始めれば、
それ以降の回答から `SURVEY_VERSION` と `question_version` に値が入る。

→ **そちらの `(question_id, question_version, segment_key)` 突合は、当面 `question_version` が空になる前提で組んでほしい**
（空を許容するか、`0` 等の既定に寄せるかはそちらの判断）。運用開始後に値が入り始める。

## 4. 【要注意】`NOTSTART` 行は構造的に出力できない

layout の `STA` に `NOTSTART` の定義はあるが、**実データには絶対に出ない**。
Hibi のエクスポートは「セッションを持つ回答者」だけを行にするため、未開始者は行として存在しない。
layout の note にもその旨を書いた。クリーニング裁定の対象は `COMP` / `PARTIAL` / `ABANDON` の3つになる。

## 5. `trait_key` について

`trait_key` 列は出したが、**中身は `metric_code`（設問メタの共通指標コード）**で、未設定の設問は空欄になる。
旧設問の「対応表で解決した値」は Hibi 側に対応表が存在しないため出せない。
ハブ辞書との突合は、新規設問に `metric_code` を付ける運用が回り始めてから埋まっていく。


- 未完了行が既定で入る。センチネルは既存どおり（`.` 無回答 / 94 システム上未到達 / 95 分岐非表示 …）。

## 8. コード表の運用ルール（そちら側の実装に影響する）

`JOB` / `BUS` のコードは **追記のみ・番号の再利用禁止**（既存出力の意味が変わるため）。
プロフィールの選択肢に無いラベル（旧データ・表記ゆれ）は **`99`** で出る。原文が要る場合は `mode=label` で出力できる。
