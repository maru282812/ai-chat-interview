# サイト構成設計：案件検索型サイトへの統合（unii-research型）

作成日: 2026-07-04
ステータス: 設計ドラフト（未実装）

## 1. ゴール

現状は「案件ごとのLIFF URL・マイページ・回答画面」が分散している。これを
**「1つのサイトの中で案件を探す→応募する→回答する→謝礼を受け取る」が完結する検索型サイト**に統合する。
参考モデルは unii-research（ユニーリサーチ）の案件一覧・詳細・マイページ構成。

## 1.5 設計原則：スマホが基本（モバイルファースト）

**このサイトの正はスマホ（LIFF）である。** ユーザーはLINE経由でしか来ない。

- すべての画面はまずスマホレイアウトで設計・実装・検証する。§5のスマホ図が正で、PC 2カラムは「あれば嬉しい拡張」にすぎない
- PC対応は Phase 4 まで着手しない。Phase 1〜3 でPC見栄えのためのコード・CSSを先回りで書かない
- 実装時の検証も Playwright のモバイルビューポート（375×812 等）を既定にする
- タップターゲット・下部ナビ・親指リーチ・縦スクロール前提のカード設計を崩さない。ホバー依存のUI（ツールチップ等）は使わない
- 将来のWeb版もスマホWebが先。デスクトップは最後

## 2. 大事にする既存の考え方（変えないこと）

| 既存思想 | 本設計での扱い |
|---|---|
| 下部ナビ4タブ（探す/保存/やりとり/マイページ）は実装済み | そのまま骨格として採用。サイトの4大セクションになる |
| 回答画面にはナビを出さない（集中妨害防止） | `/liff/survey/*` 系は引き続きナビ無しの没入画面 |
| 公開制御 = `status='published'` × `visibility_type='public'`（誤表示遮断） | 一覧・詳細・応募すべてこのフィルタを通す。店舗専用案件は絶対に混ぜない |
| 店舗専用導線 `/liff/store?entry_code=` は別入口 | 検索サイトとは独立した入口のまま。完了画面の会員化CTA→「探す」へ合流 |
| 通常案件はセルフ回答不可（assignment必須・サーバー権威） | 「応募」モデルで橋渡しする。応募≠回答開始。assignment発行は引き続きサーバー側の権威判断 |
| 同意フロー `/liff/consent` の再利用・書類整理が先（P0） | 応募ゲート・会員登録ゲートに同意チェックを配管。書類整理（14件棚卸し）と同期して進める |
| AIチャット深掘り（interview_chat）が差別化 | 案件カードの「形式」表示を AIインタビュー / アンケート で明示。人間の日程調整が不要な分、**応募→即開始**も可能（後述の auto_accept） |
| デイリー・日記・愚痴・性格は別コンテンツ | 検索対象にせず、マイページ/やりとりからの導線に留める |

## 4. 応募モデル（新規・本設計の中心）

現行の「通常案件はadmin pushでassignmentを作らないと回答できない」思想と、
検索サイトの「ユーザーが自分で応募する」体験を両立させる橋渡し。

```
ユーザー: 探す → 詳細 → 応募（事前設問+同意）
                          │
                  project_applications 作成 (status: applied)
                          │
        ┌─────────────────┴─────────────────┐
        │ 応募方式=手動選考（人インタビュー等）│ 応募方式=自動承認（AIインタビュー/アンケート）
        │ 管理者が選考                        │ スクリーニング通過なら即
        │ 当選 → assignment発行 → 通知        │ assignment発行 → そのまま回答開始
        │ 落選 → 落選通知                     │ 不通過 → その場で対象外表示
        └───────────────────────────────────┘
```

- **assignment発行は常にサーバー側**（IDOR対策・サーバー権威の思想を維持）。応募はあくまで「発行リクエスト」。
- AIインタビューは日程調整不要なので `auto_accept` により unii にない「応募→即回答」体験が作れる（差別化）。
- `max_respondents`（既存カラム）到達で募集自動クローズ（一覧に「まもなく終了」、詳細で応募ボタン無効化）。
- 月間応募数（n/10件）は project_applications の当月カウントで算出。新テーブル不要。

### DB差分（最小）

```sql
-- project_applications 🆕
id uuid PK / project_id FK / line_user_id text / respondent_id uuid null
status text check (applied|accepted|rejected|withdrawn|expired)
screening_answers_json jsonb null   -- 事前設問の回答
assignment_id uuid null             -- 当選時に発行された assignment
applied_at / decided_at / created_at / updated_at
unique (project_id, line_user_id)   -- 重複応募防止

-- projects 追加カラム 🆕
tags text[] default '{}'            -- #顔出し必須 #既婚 など
ng_conditions text null             -- NG条件（自由記述・改行区切り）
recruit_deadline timestamptz null   -- 募集期限（残り日数表示・自動クローズ）
apply_mode text default 'manual' check (manual|auto)  -- 応募方式
interview_format text null          -- オンライン/チャット等の表示用
```

既存流用: `category`（生活系/仕事系タブ）, `estimated_minutes`（所要時間）, `max_respondents`（募集人数）, `reward_points`（報酬）, `display_thumbnail_url`, `screening_config`（事前設問）。

## 6. 検索タブ（Phase 2）

- キーワード検索: `user_display_title` / タグ / 募集条件テキストに対する ilike（件数規模的にDB全文検索は不要）。
- 絞り込み: カテゴリ / 形式（AIインタビュー・アンケート）/ 報酬レンジ / 所要時間 / タグ。
- 並び替え: 新着順（既定）/ 締切が近い順（recruit_deadline）/ 報酬が高い順。
- おすすめ順（Phase 4）: プロフィール（性別・年齢・タグ）× 案件タグ・スクリーニング条件の一致度スコア。

## 6.5 LIFF統合とデイリー導線（2026-07-04 確定）

### LIFF統合の方針
- サイト化＝LIFF廃止ではない。**機能ごとに分かれたLIFF ID（SURVEY/MYPAGE/CONTACT）を1本に統合**し、サイト（一覧）をLIFF endpointにする。
- 公式LINEの全入口（リッチメニュー・新着Flex・当選通知・締切リマインド）は同一LIFFの**パス付きディープリンク**に統一。通知タップ→該当ページ直行＋下部ナビで回遊可能（単発行き止まりの解消）。
- LIFF内は liff.init→getProfile で**ログイン画面なし**（従来どおり）。LINE外Web版（LINE Login）はPhase 4。
- 当選待ちはデフォルト不存在（スクリーニング通過→同一セッションで本編continue）。`ScreeningPassAction=interview/manual_hold` 使用時のみ発生し、その場合も**単発survey LIFFは残さず**「やりとり」一覧＋通知から回答ページへ直リンクのいいとこ取りで一本化。

### リッチメニュー（6ボタン・モック p82）
```
さがす(新着) │ おすすめ │ 保存
やりとり🔴  │ マイページ │ ヘルプ
```
- 下部ナビとラベルを揃える。やりとりバッジ＝当選未回答/回答途中件数。デイリーは入れない。

### デイリーアンケート導線（確定）
- 案件検索一覧には混ぜない（別モデル・応募不要のため）。
- **入口①**: 毎日のLINE push（従来どおり公式LINE内で直接回答も可）。
- **入口②**: サイト起動時インタースティシャル（モック p80）＝今日の1問だけを大きく表示・右上×・「回答してptもらう / スキップして案件をさがす」の選ばせるUI。**1日1回のみ**・回答済み/当日×済み/今日の分なしの場合は出さず直接一覧へ。
- **入口③（確定A）**: ×した人向けに探す上部へスリムなデイリーバナー（モック p81）。回答済みなら非表示。
- 回答完了→そのまま案件サイト（探す）へ着地。

## 9. リスク・注意点

- **誤表示遮断が最優先**: 応募ページ・応募APIにも `published × public` フィルタを必ず通す（詳細ページだけでなく）。
- **重複応募**: unique(project_id, line_user_id) ＋ 再応募ポリシー（rejected後の再応募可否）を案件設定にするか要検討。
- **同意ゲート**: 応募時点でグローバル必須同意（利用規約/PP/ポイント規約）未取得なら consent へリダイレクト。書類整理が先という既定方針（P0）と順序を合わせる。
- **screening_config の二重利用**: 応募時の事前設問と回答時のスクリーニングが同一設定を共有する場合、応募で回答済みの内容を回答フローに持ち越すか（carry-forward）は設問条件制御サーバー権威化（進行中）の設計と整合を取る。
- **既存assignment運用との共存**: 応募経由と admin push 経由の assignment が混在する。assignment に発行元（application/manual）を持たせると集計・デバッグが楽。
