# 実装計画: rawdata 拡張（集計アプリ ai-report からの申し送り対応）

作成: 2026-07-13 / 対象: ai-chat-interview（Hibi）
入力: `ai-report/docs/申し送り_ai-chat-interview_rawdata拡張要請.md`
関連: `docs/plan-rawdata-export.md`（rawdata 設計）/ `ai-person/docs/handoff-ai-chat-interview-integration.md`（ハブ連携）

## 実装目的

集計アプリ（ai-report）が rawdata.csv を**正準入力**として、ウエイトバック・属性クロス・
クリーニング裁定・ペルソナハブへの actuals push まで実行できる状態にする。
`respondents_wide.csv` / `answers_long.csv` / `codebook.csv` は**凍結契約のまま一切触らない**。

## 前提（現状調査の結果）

| 申し送り項目 | 現状 | 判定 |
|---|---|---|
| `RANK` | ランクが2系統（正準 `user_ranks` / レガシー `respondents.current_rank_id`）。両方に書き込み実装あり | **要決定 → 下記で確定** |
| `AGE_BAND` | `computeAge()`（rawdataExport.ts:406）が満年齢を持つ | 粗化するだけ |
| `REGION` | `PREFECTURE_CODES` はあるが地方区分表なし | 変換表を追加するだけ |
| `TIME_SEC` | `durationSeconds()` は `completed_at` が無いと `null` | 未完了の終端定義が必要 |
| `SURVEY_VERSION` | `sessions.snapshot_id`（migration 068）が回答者ごとに存在 | **回答者ごとに正確な版が出せる** |
| 未完了行 | **すでに実装済み**。`?statuses=` クエリと analysis.ejs のチェックボックスで選択可。既定が `completed` のみ | 既定値の変更のみ |
| `JOB` / `BUS` | mypage / profile-check の固定9択・11択がラベル文字列でDBに入る | コード表を定義するだけ |
| layout 3列 | `question_id` は `variable.question_id`、`question_version` は snapshot 版数、`trait_key` は `metric_code` が実体 | 列追加のみ |

DB マイグレーションは**不要**（既存テーブルの参照のみ）。

## 確定した設計判断

### 1. RANK = `user_ranks`（正準）の現在ランク `rank_code`

ai-report / ai-person とも「取得元は Hibi 側で確定が必要」（未解決事項）としており、**先方に指定は無い**。
先方が必要とするのは値のみで、rank→重みの写像は集計アプリのレシピ設定として可変。

- **出典**: `user_ranks` → `ranks.rank_code`。無ければ `respondents.current_rank.rank_code` にフォールバック。
- **時点**: 出力時点の現在ランク（回答完了時点の復元はしない）。
- **理由**: `respondent_rank_histories` を書いているのはレガシーの `rankService`（`respondents.total_points` 基準）
  で、ポイントの正準は `user_points` に移行済み。履歴から「完了時点」を復元しても**復元元が不整合**なので
  精度が上がらない。会員がマイページで見ているランク（`user_points` 由来）と一致させる方が意味論的に正しい。
- **layout に明記**: 「出力時点の現在ランク（回答完了時点ではない）」＋ `rank_code` → `rank_name` / `min_points` / `sort_order` の対応行。
- **先方への申し送り事項**: 定点比較で過去 wave を再出力するとランクが変わりうる（非再現）。
  再現性が要る場合は export 時点を `export_jobs` の履歴で固定して扱うこと。

### 2. TIME_SEC = 未完了行も実値を出す

- 完了: `completed_at - started_at`（既存 `total_duration_sec` と同値）。
- 未完了: `max(answers.created_at) - started_at`（AI深掘り回答も含む）。回答ゼロなら空欄。
- 凍結契約の `total_duration_sec`（wide 側）は**触らない**。rawdata 側で別計算する。
- layout に定義を明記。

### 3. UserAgent / IPAddress は既定で除外

- `?includePii=1` のときだけ出力。管理画面はチェックボックス（既定OFF）。
- layout の行も出力時のみ含める（layout 駆動で読む集計アプリ側と整合する）。

### 4. JOB / BUS のコード化

- mypage（2箇所）・profile-check（1箇所）の選択肢リテラルが真実の定義。単一の定数へ寄せる。
- コード未定義のラベル（旧データ・表記ゆれ）は `99`（不明）にする。原文が要るときは `mode=label` で出す。
- **注意**: 選択肢の並びを将来変えるとコードの意味が変わる。コード表は**追記のみ・番号の再利用禁止**とする（PRE/SEX と同じ扱い）。

### 5. 未完了行の既定を変更

- `statuses` の既定を `completed` のみ → `completed,partial,abandoned` に変更。
- `not_started` は**そもそも出力できない**（`loadExportData` がセッションを持つ回答者のみ選ぶため行が存在しない）。
  layout の `STA` に定義はあるが実データには出ない旨を先方に伝える。

## 変更対象

| 領域 | 変更有無 | 内容 |
|---|---|---|
| DB | なし | migration 不要 |
| API | あり | `/rawdata.csv` に `includePii` クエリ追加・`statuses` 既定変更 |
| UI | あり | analysis.ejs にPII チェックボックス追加・ステータス既定チェック |
| 型定義 | あり | `RawdataRespondent` に `rank_code` / `snapshot_version` / `last_answer_at` |

## 実装フェーズ

### Phase 1: 純関数（rawdataExport.ts）に新列を実装
- `REGION_BY_PREFECTURE`（8地方区分・沖縄は九州）/ `JOB_CODES` / `BUS_CODES` / `ageBand()` を追加
- `writeAttributeCells` に `AGE_BAND`（AGE直後）/ `REGION`（PRE直後）/ `RANK`（属性末尾）を追加、`JOB`/`BUS` をコード化
- メタ列に `TIME_SEC`（TIME直後）/ `SURVEY_VERSION`（STA直後）を追加
- `buildRawdataRows` に `includePii` オプション（既定 false）
- `buildRawdataLayoutRows` を新列に追随＋`question_id` / `question_version` / `trait_key` 列を末尾追加
- 完了条件: `src/tests/rawdataExport.test.ts` を拡張して全新列を検証。既存テスト無変更で通る

### Phase 2: データ供給（statExportService.ts）
- `userRankRepository.listByLineUserIds()` を追加（`user_ranks` + `ranks` の一括取得）
- `questionnaireSnapshotRepository` から `session.snapshot_id` → `version` を一括解決
- `loadExportData` の `RawdataRespondent` に `rank_code` / `snapshot_version` / `last_answer_at` を詰める
  （`withProfiles` と同じく **rawdata 出力時のみ**取得し、wide/long のクエリを増やさない）
- 完了条件: `statExportService` 経由の rawdata.csv に実値が出る。wide/long/codebook のバイト列が変わらない

### Phase 3: 出力オプションとUI
- `rawdataExportOptions()` に `includePii`、`statuses` 既定を3ステータスに変更
- analysis.ejs: 「UserAgent/IPを含める（不正検出用）」チェック（既定OFF）・ステータス既定チェック
- `export_jobs` の監査ログに `includePii` を記録
- 完了条件: 既定DLで UA/IP 列が無い。チェックONで出る

### Phase 4: 受け入れ確認と返信
- 契約テスト（wide/long/codebook）が**無変更で緑**
- 完了＋未完了を含む実サンプル1式を再エクスポートして ai-report へ渡す
- 申し送りの受け入れ条件7項目をチェックして返信（`not_started` が出ない件・RANK が現在値である件を明記）

## 注意点

- **凍結契約**: `buildWideRows` / `buildLongRows` / `buildCodebookRows` に触れない。`loadExportData` の共通部分を変える際は、
  wide/long が読まないフィールドの追加に留める（`withProfiles` の既存パターンを踏襲）。
- **列順**: 新列は隣接位置に挿入する（末尾一括ではない）。集計アプリは layout 駆動で列名参照のため位置非依存。
- **PII**: 既定除外に変えると、不正検出目的で UA/IP を使っている運用があれば影響する（現状は出力のみで検出ロジックは無い）。
- **JOB/BUS**: 既存DBに選択肢外の値が入っていないか、適用前に実データを確認する。
