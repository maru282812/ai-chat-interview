# 実装計画: Freeasy水準ロウデータ出力

要件: docs/plan-rawdata-export.md（2026-07-12 確定）

## 実装目的

自社集計担当（Python/pandas）が、業界標準パネル（Freeasy）水準の情報量を持つロウデータを管理画面からダウンロードできるようにする。完成時にできること:

- 出力画面でステータス別件数（完了/途中離脱…）を見て、含める対象をチェックで選び、Freeasy式命名の `rawdata.csv` と対応表 `rawdata-layout.csv` をDLできる
- コード出力⇄回答値（ラベル）出力を切り替えられる
- 設問別回答時刻・属性列（性別/年齢/都道府県/職業/業種/未既婚/子供）・AI深掘り統合列が付く

## 前提

- 土台: `src/lib/statExport.ts` / `src/lib/codebook.ts` / `src/services/statExportService.ts`（loadExportData がセッション選択・同意フィルタ・スナップショット列順まで実装済）
- **凍結契約**: respondents_wide / answers_long / codebook の3ファイルは列変更禁止（末尾追加のみ）。本計画は凍結ファイルに触れない（新規ファイルのみ）
- マトリクス回答は `{行index: 列index}`（single）/ `{行index: [列index...]}`（multi）のオブジェクト（question_config.matrix_rows / matrix_cols 基準）で保存されている（survey.ejs 実装確認済）。**サーバー側で answer_text / normalized_answer のどちらに入るかは Phase 1 冒頭で実データ確認**
- 設問別回答時刻は `Answer.created_at` から取れる（DB変更なし）
- 属性は `user_profiles`（gender/birth_date/prefecture/occupation/industry/marital_status/has_children）。respondent.line_user_id で結合
- コンセプト(L1)は運用未開始のため初版は単一コンセプト前提（concept_code は無視。L1結線後に列展開ルールを拡張）

### ⚠️ 前提の食い違い（Phase 0 の理由）

メモリ上「実装済・本番適用済」とされていた **export_jobs 監査ログ（migration 076/077・exportJobRepository・recordStatExport）が main に存在しない**。
stash@{0}（= commit 93a7edf 相当）に adminController / http.ts / adminAuth.ts の配線差分のみ残存。新規ファイル（repository・migration）は未追跡のまま消失したとみられる。
migration は本番適用済の可能性が高い（db:migrate 実行・pg_policies 検証の記録あり）＝ **本番にテーブルだけあり、コードが書き込んでいない状態**の疑い。

## 実装フェーズ

### Phase 0: export_jobs 監査ログの復旧（独立・先行）

1. 本番DBの実態確認: export_jobs テーブルと RLS/policy の有無（Management API 経由。無ければ 076/077 を新規作成して `npm run db:migrate`）
2. `git stash show -p "stash@{0}"` から adminController / lib/http.ts / middleware/adminAuth.ts の配線を回収
3. `src/repositories/exportJobRepository.ts` を再作成（project_id / export_type / filters_json / exported_by / exported_at。**053/066の型＝RLS有効+service_roleのみポリシー**）
4. 既存統計エクスポート5ハンドラで生成成功後・レスポンス前に await で記録（Vercelサーバーレス方針: 投げっぱなし禁止）
- 完了条件: 本番で wide をDLすると export_jobs に1行入る。anon キーで読取/挿入不可

### Phase 1: 純関数層（lib/rawdataExport.ts）

依存: なし（Phase 0 と並行可）

1. **マトリクス保存形式の実確認**（answers テーブルの matrix 設問1件を目視）→ パース仕様を確定
2. `assignQNumbers(variables)`: master_order 順に q1..qN を採番。`{ variable, qNumber }` の対応を返す
3. `buildRawdataRows(variables, respondents, options)`:
   - メタ列: `MID`（respondent_key）/ `START` / `END` / `TIME`（h:m:s）/ `STA`（response_status を COMP/PARTIAL/ABANDONED 等に写像）
   - SA: `q{n}`（コード値 or ラベル）、MA: `q{n}c{k}`（0/1・one-hot ロジックは statExport の getSelected を再利用）、自由記述・その他: `q{n}t{k}`
   - マトリクス: `q{n}s{j}`（SA行）/ `q{n}s{j}c{k}`（MA行）に展開。パース不能時は `q{n}` 1列に生値＋cleaning_note
   - `q{n}_datetime`: 該当設問の primaryAnswer.created_at
   - 属性列（末尾）: `SEX/AGE/PRE/JOB/BUS/MAR/CHI`。コード化は layout に定義（gender: male=1/female=2/other=3/prefer_not_to_say=9、AGE=回答時点の満年齢、PRE=JIS都道府県コード、JOB/BUS はマスタ化されていなければラベルのまま出力し layout に「コード未定義」と明記）
   - options: `mode: "code"|"label"` / `statuses: string[]`（含めるステータス） / `includeProbe: boolean`（既定 true・`{code}_final_answer_text` 等を付加） / `excludeTest`
4. `buildRawdataLayoutRows(assignments)`: q番号 / question_code / 設問文 / 列名 / 選択肢コード / ラベル / 型 / 欠損コード / 属性コード表
5. `buildStatusCounts(respondents)`: response_status 別件数（テスト回答は別掲）
6. テスト `src/tests/rawdataExport.test.ts`: 採番順序・SA/MA/マトリクス/その他テキスト展開・code/label切替・ステータスフィルタ・属性コード化・欠損センチネル
- 完了条件: 新テスト pass・既存 statExport.test.ts green・typecheck green

### Phase 2: サービス・ルート結線

依存: Phase 1

1. `statExportService` に追加（loadExportData を共有）:
   - `rawdataCsv(projectId, options)` / `rawdataLayoutCsv(projectId)` / `statusCounts(projectId, options)`
   - loadExportData 拡張: userProfileRepository で line_user_id 一括取得（一括メソッドが無ければ repository に追加）→ ExportRespondent に `profile?: {...}` を付与（optional・wide/long は読まないため凍結契約に影響なし）
   - ステータスフィルタ: 現状 loadExportData はセッション有り回答者のみ対象。`statuses` は mapResponseStatus 後の値でフィルタ
2. ルート: `GET /admin/projects/:projectId/exports/stat/rawdata.csv` / `rawdata-layout.csv` / `status-counts`（JSON）
3. ハンドラ: statExportOptions を拡張（`mode` / `statuses`（CSV区切り・既定 completed）/ `includeProbe`）。**Phase 0 の export_jobs 記録を rawdata にも適用**
- 完了条件: curl で rawdata.csv / layout がDLでき、`?mode=label&statuses=completed,partial` が効き、export_jobs に記録される

### Phase 3: 出力画面カード

依存: Phase 2

   - ステータスごとのチェックボックス（**既定 = 完了のみON**）＝「いれるいれないは手動判断だが、画面にわかる形で」

2. モバイル崩れがない素朴なHTML（既存カードのスタイル踏襲）
- 完了条件: 画面で件数が見え、チェック変更がDL内容に反映される

## 注意点

- **凍結契約**: wide/long/codebook のビルダー・列に触らない。ExportRespondent 型への追加は optional のみ
- 監査ログはレスポンス前 await（Vercelサーバーレス方針・投げっぱなし禁止）
- 属性は個人情報。rawdata にも line_user_id 等の直接識別子は出さない（§19 踏襲・respondent_key のみ）
- occupation/industry が自由文字列の場合のコード化は無理にしない（ラベル出力＋layoutに明記）
- q番号はスナップショット確定後に安定する。スナップショット未確定プロジェクトでは現行マスター順採番となることを layout ヘッダに注記
